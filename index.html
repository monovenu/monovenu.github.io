jQuery选择器

博主最近重温了jQuery的选择器，再次感到了它的强大。

1、基本选择器

$(‘选择器’)。基本选择器通过id, class或元素名来选择。
多选$(‘span,#one’)

2、层次选择器

通过DOM元素之间的层次关系来选择。
后代, $(‘div span’)
子元素, $(‘div>span’),不包含孙子辈及以下的
相邻下一个兄弟, $(‘.one+div’)
之后所有兄弟, $(‘.one~div’)

3、过滤选择器

与css的伪类选择器语法相同，以”:”开头。

基本过滤选择器

第一个, :first, $(‘div:first’)选取所有div中的第一个
最后一个, :last, $(‘div:last’)
去除匹配的元素, :not(选择器), $(‘div:not(:first)’)选取所有div,除了第一个div.
注意:not()内部没有引号。
偶数号索引, :even $(‘input:even’), 索引从0开始
奇数号索引, :odd $(‘input:odd’)
索引等于index, :eq(index), $(‘input:eq(1)’)
索引大于index, :gt(index), $(‘input:gt(2)’)
索引小于index, :lt(index), $(‘input:lt(2)’)
获得焦点, :focus, $(‘focus’)

内容过滤选择器

依据子元素或文本内容进行过滤
选择含有文本内容的元素, :contains(‘文本’), $(“div:contains(‘我’)”)

可见性过滤选择器

可见, :visible, $(‘:visible’)
不可见， :hidden

属性过滤选择器

依据css属性选择器，如$(‘[title=test]’)选取属性为test的元素

表单属性过滤选择器

对表单元素进行过滤。
可用, :enabled, $(‘input:enabled’)
被选中, :checked
被选中的选项,用于下拉列表 :selected, $(‘select option:selected’)

HTTP状态响应码与nginx

HTTP状态响应码表示服务器对于客户端的HTTP请求的响应状态。

1、1xx 消息

代表请求已被接受，需要继续处理。

2、2xx 成功

200 OK

请求成功

201 Created

请求的资源已经被创建

202 Accepted

请求被接受，但服务器还未进行进一步的操作。不能保证最终请求会成功。

3、3xx 重定向

这类状态码代表需要客户端使用替代地址访问资源。后续的请求地址（重定向目标）在本次响应的 Location 域中指明。

300 Multiple Choices

被请求的资源有一系列可供选择的版本，每个都有自己特定的地址。用户或浏览器能够自行选择一个首选的地址进行重定向。
除非这是一个 HEAD 请求，否则该响应应当包括一个资源特性及地址的列表的实体，以便用户或浏览器从中选择最合适的重定向地址。这个实体的格式由 Content-Type 定义的格式所决定。浏览器可能根据响应的格式以及浏览器自身能力，自动作出最合适的选择。
如果服务器本身有了首选地址，在 Location 中应当指明首选的 URI。

301 Moved Permanently

永久重定向。被请求的资源已永久移动到新位置。
新的永久性的 URI 应当在响应的 Location 域中返回。除非这是一个 HEAD 请求，否则响应的实体中应当包含指向新的 URI的超链接及简短说明。

302 Found

临时重定向。请求的资源现在临时从不同的 URI 响应请求。由于这样的重定向是临时的，客户端应当继续向原有地址发送以后的请求。

303 See Other

类似于302. 用于重定向POST请求到一个新的资源。而且客户端应当采用 GET 的方式访问那个资源。

304 Not Modified

资源未修改，从浏览器缓存中获取资源。

305 Use Proxy

被请求的资源必须通过指定的代理才能被访问。 Location 域中将给出指定的代理所在的 URI 信息。

307 Temporary Redirect

类似于302。

4、4xx 请求错误

这类的状态码代表了客户端可能发生了错误。

400 Bad Request

语义有误或者请求参数有误。
在nginx中的400错误是因为请求报文实体过大，实体过大，通常是由于cookie中写入了较大的值所引起。解决方法是在nginx.conf中，将client_header_buffer_size和large_client_header_buffers都调大。

401 Unauthorized

当前请求需要验证客户端。该响应必须包含一个适用于被请求资源的 WWW-Authenticate 信息头用以询问用户信息。客户端可以重复提交一个包含恰当的 Authorization 头信息的请求。
如果当前请求已经包含了 Authorization 证书，那么401响应代表着服务器验证已经拒绝了那些证书。
比如在Nginx下对网站进行密码保护。可以更改网站的Nginx的server配置，创建htpasswd文件保存密码，它和nginx.conf在同一级目录。

403 Forbidden

服务器已经理解请求，但是拒绝执行它。
nginx出现403 forbidden最常见的原因是权限配置不正确。
为了保证文件能正确执行，nginx既需要文件的读权限,又需要文件所有父目录的可执行权限。
例如，当访问/usr/local/nginx/html/image.jpg时，nginx既需要image.jpg文件的可读权限，
也需要/,/usr,/usr/local,/usr/local/nginx,/usr/local/nginx/html的可以执行权限。
解决办法:设置所有父目录为755权限，设置文件为644权限可以避免权限不正确。

404 Not Found

请求失败，url错误，请求所希望得到的资源未被在服务器上发现。
NGINX可以自定义404页面。方法是
1.创建自己的404.html页面
2.更改nginx.conf在http定义区域加入： fastcgi_intercept_errors on;
3.更改nginx.conf中在server区域加入： error_page 404 /404.html 或者 error_page 404 =http://www.xxx.com/404.html
4.更改后重启nginx,，测试nginx.conf正确性： /opt/nginx/sbin/nginx –t

5、5xx 服务器错误

这类状态码代表了服务器在处理请求的过程中有错误发生，也有可能是服务器意识到以当前的软硬件资源无法完成对请求的处理。

500 Internal Server Error

服务器遇到了一个未曾预料的状况，导致了它无法完成对请求的处理。
nginx下出现500错误可能是硬盘空间满了或者nginx配置文件错误。

AMD

AMD是”Asynchronous Module Definition”的缩写，意思就是”异步模块定义”。
它采用异步方式加载模块，模块的加载不影响它后面语句的运行。
所有依赖这个模块的语句，都定义在一个回调函数中，等到加载完成之后，这个回调函数才会运行。
AMD也采用require()语句加载模块，但是不同于CommonJS，它要求两个参数：

第一个参数[module]，是一个数组，里面的成员就是要加载的模块；第二个参数callback，则是加载成功之后的回调函数。

math.add()与math模块加载不是同步的，浏览器不会发生假死。所以很显然，AMD比较适合浏览器环境。
目前，主要有两个Javascript库实现了AMD规范：require.js和curl.js。

require.js

require.js的作用

最初，所以js代码都写在一个文件中，只要加载这一个文件就够了。后来，代码越来越多，必须分成多个文件，依次加载。

这样的写法存在两个问题。一是，加载的时候，浏览器会阻塞，停止渲染，加载文件越多，网页失去响应的时间就会越长。二是，因为Js文件之间存在依赖关系，所以严格保证加载顺序。当依赖关系很复杂时，代码的编写和维护都会变得很困难。require.js的诞生，就是为了解决这两个问题：

主模块的写法

上一节的main.js，我把它称为”主模块”，意思是整个网页的入口代码。它有点像C语言的main()函数，所有代码都从这儿开始运行。

但这样的话，就没必要使用require.js了。真正常见的情况是，主模块依赖于其他模块，这时就要使用AMD规范定义的的require()函数。

require()函数接受两个参数。第一个参数是一个数组，表示所依赖的模块，上例就是[‘moduleA’, ‘moduleB’, ‘moduleC’]，即主模块依赖这三个模块；第二个参数是一个回调函数，当前面指定的模块都加载成功后，它将被调用。加载的模块会以参数形式传入该函数，从而在回调函数内部就可以使用这些模块。
require()异步加载moduleA，moduleB和moduleC，浏览器不会失去响应；它指定的回调函数，只有前面的模块都加载成功后，才会运行，解决了依赖性的问题。

CommonJS规范

博主最近在学习模块化，看完阮一峰写的博文和Samaritans写的前端模块化后获益匪浅，
现整理后转载如下：

模块化的最大好处就是代码重用。我们可以使用别人的代码，
想要什么功能，就加载什么模块。
但是，这样做的前提是大家以同样的方式编写模块。
否则你有你的写法，我有我的写法，岂不是乱了套！
考虑到Javascript模块现在还没有官方规范，这一点就更重要了。

1、node.js

2009年，美国程序员Ryan Dahl创造了
node.js项目，将javascript用于服务器端编程。
这标志”Javascript模块化编程”正式诞生。
因为老实说，在浏览器环境下，没有模块也不是特别大的问题，毕竟网页程序的复杂性有限；
但是在服务器端，一定要有模块，与操作系统和其他应用程序互动，否则根本没法编程。

node.js的模块化，遵从CommonJS规范。

2、CommonJS

定义模块

根据CommonJS规范，一个文件就是一个模块。每个模块都是一个单独的作用域，也就是说在该模块内部定义的变量，无法被其他模块读取，除非定义为全局对象的属性。

模块输出

模块只有一个出口，module.exports对象。

加载模块

使用require方法读取文件并执行，返回module.exports对象。
实例如下：

//定义模块 myModel.js
var name="lyx";
function printName(){
    console.log(name);
}
module.exports={
    printName: printName
}

//加载模块
var nameModule=require('myModule.js');
nameModule.printName();

3、浏览器环境

有了服务器端模块后，大家自然想要客户端模块。
如果两者能够兼容，一个模块不用修改就可以在服务器和浏览器上运行。
然而，看下面的代码。

var nameModule=require('myModule.js');
nameModule.printName();

执行nameModule.printName();必须要等myModule模块加载完成。
如果加载时间很长，整个程序就会等待。这对服务器端不是问题，
因为所以模块存放在本地硬盘，可以同步加载，等待时间就是磁盘读取时间。
但是，对于浏览器，这却是一个大问题，因为模块都放在服务器端，
等待时间取决于网速的快慢，可能要等很长时间，浏览器处于”假死”状态。
因此，浏览器端的模块，不能采用”同步加载”（synchronous），
只能采用”异步加载”（asynchronous）。这样AMD规范就因运而生了。

清除浮动造成的影响

本文参考了知乎问题以及对overflow与zoom”清除浮动”的一些认识。

当在一个div中浮动了一个子元素时，该div会塌陷，
不会扩充到浮动子元素的高度。解决这个问题主要有4种方式。

1、clear方式

添加空标签，给空标签设置clear属性。

优点是简单易行。缺点是增加了无意义的空标签，不利于语义化。

2、给父级添加overflow

优点是不存在结构和语义上的问题，缺点是内容增多时候容易造成不会自动换行导致内容被隐藏掉。

BFC

为什么给父级添加overflow会起作用呢。要解释这个问题先要了解BFC.BFC的全称是块级排版上下文。也可以称之为块布局上下文。
每个视口会默认建立一个块布局上下文。假设一个文档所有的块盒overflow属性都是”visible”，那么它们必定属于一个块布局上下文，也就是视口所产生的块布局上下文。但是一旦有一个块盒设置了”visible”以外的值，新的块布局上下文就产生了。它就像为它的内容开辟了新的摆放空间，这个摆放空间是相对独立的。
这个独立的块布局上下文会产生一种“包裹”效果，父元素会刚好包裹住子元素，所以overflow: hidden的父元素不会塌陷。
会产生独立的块布局上下文的情况还包括：浮动元素、绝对定位元素、除块盒(block)以外的块容器(如table-cells)。

3、父级float

原理同overflow, 会形成独立的块布局上下文。
优点是不存在结构和语义上的问题，缺点是与父元素相邻的元素的布局会受到影响。

4、::after伪元素

给父类添加伪元素。

该方式与clear方式原理一致，但是不会添加空标签。
::after会在元素的内容后面附加内容，将添加的内容设置为块级显示并clear就可以清除浮动造成的影响。
:: 表示伪元素，: 表示伪类。为兼容老版IE,也用:after。

总结

实际上，以上的方式可以归类为2种。一种是清除浮动方式：clear.另一种是形成独立的块布局上下文，让父元素包裹子元素。
衡量利弊后，博主认为第四种方式适用性更好。

Ajax跨域方法总结

同源策略

为了安全，js脚本不能访问其他服务器的数据。
具体来说就是，域名、端口号、协议有一个不同就不能访问。

跨域

做ajax读取数据的时候，经常会遇到ajax需要跨域的问题。跨域就是实现不同域名之间的通信，但对端口号和协议不同的情况无法解决。

跨域方法

1、img标签的开放策略

一个网页可以从任何网页加载图像，不存在跨域问题。
可以动态创建图像，侦听load事件来确定是否接收到响应。

这种方式的缺点是只能客户端向服务器端发送数据，客户端只能得到
服务器端返回的图片，得不到具体数据。所以通常用于统计点击页面次数。

2、JSONP

script类似于img,它的src属性也有开放策略，可以将数据放在其他域名的
js文件里，通过script标签加载js文件来访问数据。
在页面中创建script标签向不同域请求数据的方法就是JSONP.
假设在http://example1.com/index.php这个页面中向http://example2.com/getinfo.php提交GET请求，
我们可以将下面的JavaScript代码放在http://example1.com/index.php这个页面中来实现：

当GET请求从http://example2.com/getinfo.php返回时，可以返回一段JavaScript代码(JSONP)，这段代码会自动执行，可以用来负责调用http://example1.com/index.php页面中的一个callback函数。
jQuery中实现JSONP的方式如下：
首先在客户端注册一个callback,然后把callback的名字(如:jsonp1)传给服务器。
注意：服务端得到callback的数值后，要用jsonp1把将要输出的json内容包括起来，此时，服务器生成 json 数据才能被客户端正确接收。
然后以javascript语法的方式，生成一个function， function 名字就是传递上来的参数 ‘jsoncallback’的值 jsonp1.
最后将json数据直接以入参的方式，放置到 function 中，这样就生成了一段 js 语法的文档，返回给客户端。
客户端浏览器，解析script标签，并执行返回的 javascript 文档，此时javascript文档数据，作为参数， 传入到了客户端预先定义好的 callback 函数。

3、CORS

W3C提供了跨域请求的标准方案（Cross-Origin Resource Sharing）。
IE8、Firefox 3.5+ 、Chrome浏览器、Safari 4 等已经实现了CORS规范。
在服务器响应客户端的时候，带上Access-Control-Allow-Origin头信息。
如果设置 Access-Control-Allow-Origin:*，则允许所有域名的脚本访问该资源。
Access-Control-Allow-Origin:http://www.baidu.com/允许特定的域名访问。
具体实现可以在Php中设置header()

遗憾的是IE8和IE9支援CORS要通过XDomainRequest对象,IE10开始则提供了原生支持。
